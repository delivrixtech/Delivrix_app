# Fase 0.5 Result — Sign Endpoint + Dispatcher

Fecha de ejecucion: 2026-05-29.
Owner: Codex backend.

## Resultado

- Implementado `POST /v1/openclaw/proposals/:proposalId/sign`.
- Implementado `POST /v1/openclaw/proposals/:proposalId/reject`.
- Implementado `GET /v1/openclaw/proposals/:proposalId/status`.
- Implementado dispatcher interno de skills con adapter HTTP in-memory.
- Implementada matriz canonica `skillSlug -> actionIds` para impedir que una propuesta firme un handler distinto al action aprobado.
- Gateway fail-closed en produccion para `/sign` y `/reject`: requiere HMAC salvo modo local unsigned panel sobre loopback.
- El modo local unsigned panel exige opt-in explicito `OPENCLAW_SIGN_ALLOW_UNSIGNED_LOCAL_PANEL=true`, loopback real y `Origin` local de Vite.
- Kill switch revalidado justo antes de invocar el dispatcher interno.
- Timeouts dejan proposal en `executing` y registran finalizador in-process cuando el handler termina.
- Summaries de handler redaccionan secretos en objetos y strings.
- El `signatureId` publico (`sig_*`) queda separado del approval token privado; los handlers internos validan el token por `approvalTokenHash` y Canvas/status/audit publico no exponen el token.
- Los handlers live ya no aceptan aprobaciones legacy basadas solo en `executionId`; si no hay `approvalTokenHash`, no hay autorizacion ejecutable.
- Re-review QA con sub-agente senior: sin blockers P0/P1 despues del cierre.
